# Reproducing the report

Each section below maps a section of the PDF report to the script(s) and command(s) that produce its numbers. All commands are run from the repo root.

> **Setup:** `pip install -r requirements.txt`. No API keys are required for any public providers used.

### Two gaps where verification needs an external source — both defensible

**Gap 1 — Live paper-trading P&L (V1 −96.4%, V2 −1.0% / −17.8%).**
These come from the live trader's state files (`Data/paper_trading/trades.json` for V1, `Data/paper_trading_v2/trades.json` for V2), which are runtime-generated logs, not derived from any model or backtest. They are intentionally **not** shipped here for two reasons: (a) they are operational artifacts of a live trading session that cannot be regenerated by re-running code; (b) the verifiable claim of the report is that the *backtest predicts the live result*, and the backtest itself **is** fully reproducible above. The live numbers can be cross-checked against on-chain Polymarket transaction history if needed (the V1 / V2 wallet addresses are documented in the live system source).

**Gap 2 — `retro_backtest_late_season.py` (Apr 6–13 retro on FAV-only filter).**
This script calls the live Polymarket CLOB price-history endpoint, which retains **only the most recent 30 days** of tick data. The Apr 6–13 window is currently in-window, but a rerun more than 30 days after Apr 13 will return empty for the earliest dates and produce a slightly different bet count and P&L. This is a structural limitation of the upstream API, not a bug in the script. The relevant point in the report — that the 8-day post-pause window is consistent with the late-March regime (5 of 6 bets losing, one Boston outlier) — is derived from the simulation itself and reproduces as long as PM still serves the relevant tokens. For permanent verification, the originally-produced output `Data/backtest/analysis/retro_late_season_trades.csv` was generated on Apr 17 and is preserved in the parent project's analysis folder.

### Requires DB rebuild (large, gitignored, optional)

Retraining the model from scratch and rerunning live predictions both depend on `Data/TeamData.sqlite` and `Data/OddsData.sqlite` (~hundreds of MB combined, NBA-Stats and SBR backfill). They are **not required** to verify any number in the report, because the backtest CSVs in `Data/backtest/` already contain the model's predictions. The DB rebuild commands are documented below in case Dr. Yang wants to retrain from raw inputs.

## 1. The trained model — XGBoost ML, 69.8% (Table 2)

The model used throughout the report:
`Models/XGBoost_Models/XGBoost_69.8%_ML_md4_eta0p033_*.json`
(trained Jan 25 2026 on 2012–2025 seasons.)

To retrain (requires the full team & odds DBs — see step 1a):

```bash
# 1a. Rebuild training DBs (one-time, ~1–2 hours of API scraping)
python -m src.Process-Data.Get_Odds_Data --backfill
python -m src.Process-Data.Get_Data --backfill
python -m src.Process-Data.Create_Games

# 1b. Train the calibrated XGBoost classifier
cd src/Train-Models
python -m XGBoost_Model_ML --dataset dataset_2012-26 --trials 100 --splits 5 --calibration sigmoid
```

The committed model produces the **71.8% accuracy on the 2025-26 holdout** quoted in Table 3. To verify without retraining, see step 4.

## 2. The backtest dataset — Polymarket prices + ESPN WP + Q1 scores

All data needed for the report's backtest tables is already committed under `Data/backtest/`. To regenerate from scratch:

```bash
# Polymarket events + CLOB price history + XGBoost predictions
python build_backtest_dataset.py            # → Data/backtest/nba_backtest_dataset.csv

# ESPN Win Probability for the same games
python build_espn_wp.py                     # → Data/backtest/espn_wp_backtest.csv

# Q1 scores (NBA Stats API)
python fetch_q1_scores.py                   # → Data/backtest/q1_scores.csv

# Polymarket CLOB tick-level price history
python build_price_history.py               # → Data/backtest/nba_price_history.csv
```

## 11. Live system (V2 dual-leg paper trader)

The live system is not exercised at submission time, but the code that ran during the live forward test is `src/Polymarket/paper_trader_v2.py` and `src/Polymarket/scheduler.py`. To dry-run it:

```bash
python -m src.Polymarket.paper_trader_v2 --status
python -m src.Polymarket.paper_trader_v2 --init      # places bets pre-game
python -m src.Polymarket.paper_trader_v2 --monitor   # ESPN WP exits for DOG leg
python -m src.Polymarket.paper_trader_v2 --resolve   # closes finished games
```

Or the full scheduler loop:

```bash
python -m src.Polymarket.scheduler
```
